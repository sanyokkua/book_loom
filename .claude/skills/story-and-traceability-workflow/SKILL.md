# Story and Traceability Workflow

Stories are the unit of tracked work. The specification under `docs/specification/` is
frozen; stories translate frozen clauses into implementable, testable slices, and the
traceability chain proves every clause is covered by a test.

## When to use

- Creating a new story file `docs/stories/story-NNN-<slug>.md` for a phase item.
- Editing a story that is still `draft`, `ready`, or `in-progress`.
- Wiring acceptance criteria to the tests that prove them.
- Transitioning a story's status, or checking traceability before "done".
- Updating `docs/stories/README.md` (the story index) after adding/moving a story.

## When NOT to use

- Do NOT edit a story whose status is `done` — done stories are immutable. To change
  scope, author a new story that supersedes it and set the old one to `superseded`.
- Do NOT edit files under `docs/specification/` — the spec is frozen; a story consumes
  clauses, it never changes them. If a clause is wrong or missing, raise it separately.
- Do NOT hand-edit `docs/traceability.yaml` — it is generated by `./gradlew trace`.
- Do NOT plan a whole phase's stories here — use the `/plan-phase-stories-creation`
  command. Do NOT plan one story's implementation here — use
  `/plan-user-story-implementation`.

## Workflow

1. **Pick the next ID.** Scan `docs/stories/` for the highest `STORY-NNN` and assign the
   next monotonic integer. IDs are permanent and never reused, even if a story is later
   superseded. AC ids are `STORY-NNN-AC-N`, also permanent.
2. **Load binding inputs.** Read every spec clause the story implements (cite as
   `docs/specification/<path>#<anchor>`, resolved via
   `docs/specification/00_Foundation/05_SPEC_INDEX.md`), the applicable design decisions
   (`00_Foundation/04_DESIGN_DECISIONS.md`), and any ADRs under `docs/adr/`.
3. **Write the front-matter** (keys, exact order):
   `id, title, status, spec_clauses[], modules[], acceptance_criteria[], edge_cases[],
   depends_on[], adrs[], phase, owner, estimate`.
   - `status` ∈ `draft | ready | in-progress | done | superseded`.
   - `modules[]` use Gradle module + package, e.g. `:document/ua.bookloom.document.epub`.
   - `phase` is the full `PHASE_NN_SLUG` (e.g. `PHASE_01_DOCUMENT_MODEL`) and must resolve
     to `docs/implementation_plan/phases/<phase>.md` (`traceCheck` check #13).
   - `owner` ∈ `architect | coder | tester`.
   - `estimate` ∈ `S | M | L` (S: one module, a handful of files, < ~1 day;
     M: one module + tests + a port, ~1-2 days; L: multi-module or a new seam).
   - `edge_cases[]` list `EC-<AREA>-N` ids; every listed EC must appear in some AC's
     proving test.
4. **Write the body in fixed order:** Goal · In scope · Out of scope · Spec inputs ·
   Design constraints · Acceptance criteria · Test plan · Definition of done.
5. **Author acceptance criteria**, one value per AC, no implementation detail. Each is a
   `### STORY-NNN-AC-N` heading using a pattern P1..P6:
   - P1 Given/When/Then · P2 State transition · P3 Contract/API shape (Result envelope +
     ErrorCode) · P4 Rendering/control-state assertion (TestFX for JavaFX) ·
     P5 Guard/negative (pair with an `EC-` id) · P6 Visual reference (UI only, match
     `docs/specification/mockups/ui-mockup.html` for the named screen/state/theme).
6. **Write the test plan** as a table mapping each AC to: tier · test file · test method ·
   `Proves: STORY-NNN-AC-N`. The proving test carries `// Proves: STORY-NNN-AC-N` on its
   first comment or name line so the trace generator can bind it.
7. **State the traceability delta** the story creates (which clauses/ECs/modules it newly
   covers) so the reviewer can confirm zero orphans. Phase files carry a machine-readable
   **phase→clause manifest** (`phase_clauses:` YAML front-matter) that `traceCheck` reads:
   at **phase exit** (all the phase's stories `done`), every clause in the manifest must be
   cited by some non-superseded story.
8. **Set status via the lifecycle:** `draft` (being authored) -> `ready` (front-matter
   complete, ACs frozen, `depends_on` all `done`) -> `in-progress` (implementation started)
   -> `done` (DoD met) -> `superseded` (replaced by a newer story). Never skip to `done`.
9. **Before "done", run traceability:** `./gradlew trace` (regenerate
   `docs/traceability.yaml`) then `./gradlew traceCheck` (validate: zero orphans, record
   fresh). Both must pass. Until the scaffold story implements these tasks, validate by
   review against the schema `stories{}, clauses{}, edge_cases{}, modules{}`.
10. **Update the index** `docs/stories/README.md` with the new/changed story row.

- `docs/stories/STORY_TEMPLATE.md` — the full front-matter + body skeleton to copy.
- `docs/implementation_plan/05_ACCEPTANCE_CRITERIA_PATTERNS.md` — worked P1..P6 patterns.
- `docs/implementation_plan/03_TRACEABILITY.md` — the `traceability.yaml` schema, the
  phase→clause manifests, and the orphan checks.
- In-repo authorities: `docs/specification/00_Foundation/05_SPEC_INDEX.md` (clause index),
  `docs/specification/04_Build_and_Release/02_QUALITY_GATES.md` (DoD gates),

## Mandatory validation checklist

- [ ] `STORY-NNN` is unique and higher than every existing id; IDs unchanged on edits.
- [ ] Front-matter has all keys in order; `status`, `owner`, `estimate` use legal values.
- [ ] Every `spec_clauses[]` entry resolves via `05_SPEC_INDEX.md` to a real anchor.
- [ ] `phase:` is a full `PHASE_NN_SLUG` resolving to a real phase file.
- [ ] The DoD includes the implicit **clean-gate AC**: `./gradlew clean build check
      spotlessCheck` green across the whole project, no pre-existing-failure exemption.
- [ ] Every AC is atomic, one value, no implementation detail, tagged P1..P6.
- [ ] Every AC has a proving test row with a `Proves: STORY-NNN-AC-N` tag.
- [ ] Every `edge_cases[]` id appears in some AC's proving test.
- [ ] `depends_on[]` stories are all `done` before this story is `ready`.
- [ ] `./gradlew trace` regenerated the record and `./gradlew traceCheck` passes with
      zero orphans and a fresh record before status -> `done`.
- [ ] Story is NOT edited if already `done`; `docs/stories/README.md` updated.

## Gotchas

- The traceability.yaml is GENERATED — editing it by hand is the most common mistake; it
  will be overwritten and `traceCheck` will report the record as stale.
- A `Proves:` tag with a typo silently orphans the AC; the tag must match the AC id
  exactly (case and hyphens).
- "One value per AC" — an AC asserting two things cannot be cleanly proven; split it.
- A UI AC without a P6 visual-reference line against the mockup will fail the DoD.
- Marking `done` before `depends_on` stories are `done` breaks the dependency invariant;
  the planner rejects it.
