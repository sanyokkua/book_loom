# Glossary

This glossary defines the vocabulary used across the specification. Terms are the canonical names; requirement and story
text must use them consistently.

## document-model-terms {#document-model-terms}

| Term                         | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
|------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Segment                      | The smallest translatable unit of text extracted from the document — typically one paragraph, heading, list item, table cell, or verse line. Each segment has a stable id, a source-language string, an optional target string, a status, and a source hash.                                                                                                                                                                                                                                                                                                         |
| Chunk                        | A working batch of one or more whole segments packed together up to the token budget and sent to the model in a single translate call. Segments are never split across chunks; oversized single paragraphs are sentence-split only on overflow.                                                                                                                                                                                                                                                                                                                      |
| Skeleton                     | The full structural representation of the document (element tree, attributes, images, fonts, IDs, order) with translatable text nodes removed and replaced by segment references. The skeleton is never sent to the model and is never regenerated; only its text nodes change. This is what makes structure preservation correct by construction.                                                                                                                                                                                                                   |
| Inline masking / placeholder | The replacement of inline markup, locked terms, URLs, standalone/typographic numerals and numerals inside locked terms (prose numerals stay translatable), inline code and inline MathML (DD-49), and other protected spans inside a segment with opaque placeholder tokens of the form `⟦gN⟧` before translation, so the model cannot alter them. After translation the placeholders are unmasked back to the original spans, and the placeholder multiset is validated as a hard gate.                                                                             |
| Protected content            | Code, math, and technical markup preserved verbatim and never translated (DD-49): inline `<code>` and inline MathML are masked as single atomic placeholders with their text kept exactly; block code listings (`<pre>`/`<pre><code>`, fenced/indented Markdown blocks) and block-level MathML are **non-translatable blocks** — they yield no segments, are excluded from segmentation and the chunk token budget, and live only in the skeleton. Index-term/cross-reference anchor ids and targets are preserved while their visible link text stays translatable. |
| Round-trip fidelity          | The property that parsing a document to skeleton + segments and reassembling it without changing any target text yields a **structure-and-text-preserving (canonical-equal)** result: structure, nesting, IDs, images, fonts, encoding, and all text are preserved, but exact bytes may differ under re-serialization (modulo intentional language-metadata updates; TXT compares exact bytes). Verified by the round-trip golden test, which compares canonicalized output to canonicalized source (DD-43).                                                         |

## consistency-and-memory-terms {#consistency-and-memory-terms}

| Term                                    | Definition                                                                                                                                                                                                                                                                                                                                                  |
|-----------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Glossary / term dictionary              | The editable table of names and terms with their locked target renderings, type, and grammatical gender. Used to enforce terminology; can hard-lock a term via placeholder substitution. Also called the name/term dictionary.                                                                                                                              |
| Translation memory (TM)                 | A store of previously translated segments keyed by source and surrounding context, reused to keep repeated content consistent. Match tiers: **exact** (identical source — a hint, not a force), **context** (source and neighbouring segments both match — eligible for automatic reuse), **fuzzy** (near-identical source — offered as a suggestion only). |
| Rolling summary                         | A running bilingual summary of the book so far, updated at the end of each chapter/unit and injected into later prompts to preserve plot, tone, and terminology context without resending the whole book.                                                                                                                                                   |
| Deferred resolution / backward revision | A register of decisions that cannot be settled with only the text seen so far (e.g. a name whose gender is revealed later). Deferred items are resolved in an optional backward-revision pass using whole-book facts, re-rendering affected earlier segments to `REVISED`.                                                                                  |
| Quality dial                            | A single "quality vs speed" control (Fast / Balanced / Max) that owns **mechanics only**: chunk size, preceding-target block count, repair budget N, whether the judge runs, and whether backward revision runs. It never sets the trust threshold τ (DD-45).                                                                                               |
| Review-mode dial                        | The Unattended / Assisted / Manual review control. It is the **sole owner of the trust threshold τ**; a manual Settings τ is an advanced override of highest precedence (DD-45).                                                                                                                                                                            |
| Trust threshold τ                       | The minimum deterministic confidence a translated chunk must reach to be accepted automatically. Owned by the review-mode dial (never by the quality dial). Acceptance = hard gates pass ∧ confidence ≥ τ ∧ (judge off ∨ judge score ≥ τ_judge, where τ_judge defaults to τ); chunks that fail enter self-heal and, if still failing, are flagged (DD-45).  |

## pipeline-quality-terms {#pipeline-quality-terms}

| Term          | Definition                                                                                                                                                                                                                                                                                                                               |
|---------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| InferenceGate | The single-flight guard that serializes inference requests so a local model serves one request at a time, preventing concurrent calls from overloading a single-GPU backend.                                                                                                                                                             |
| Judge         | The LLM-as-judge step that scores a draft that has already passed deterministic checks. It returns `{score, verdict}`; `score ≥ τ_judge` decides (τ_judge defaults to τ), while `verdict` is advisory/logging only (DD-45).                                                                                                              |
| Self-heal     | The repair stage that fires only on a quality failure. **Directed fix** is a single call that injects concrete QA findings and asks the model to correct exactly those. **Reflect → improve** is a two-call path used when the failure is a vague quality concern with no concrete finding, optionally followed by a monolingual polish. |

## provider-terms {#provider-terms}

| Term                                   | Definition                                                                                                                                                                                                                                                  |
|----------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Provider                               | A configured LLM backend the user can translate against. Modelled as a `Provider` port with two client implementations selected by kind — an Ollama-native client and an OpenAI-compatible client — plus per-kind profile data (DD-10, DD-32).              |
| Provider profile                       | The per-kind data record (endpoint shape, auth scheme, discovery method, capability quirks) for a provider kind, selected by a factory map.                                                                                                                 |
| Ollama / LM Studio / OpenAI-compatible | Supported provider kinds. Ollama and LM Studio are local runtimes exposing OpenAI-compatible chat plus native endpoints; "OpenAI-compatible" is any endpoint speaking the OpenAI chat schema.                                                               |
| num_ctx                                | The model context-window size (in tokens). Ollama honours it reliably only on its native API, which is why Ollama uses the dedicated Ollama-native client (setting `num_ctx` via native `options`) rather than the OpenAI-compatible client (DD-10, DD-32). |

## format-terms {#format-terms}

| Term  | Definition                                                                                                                                        |
|-------|---------------------------------------------------------------------------------------------------------------------------------------------------|
| EPUB  | A ZIP-packaged XHTML e-book format (versions 2 and 3 supported). Contains an OPF package document, a spine, content documents, images, and fonts. |
| FB2   | FictionBook 2 — a single XML document (optionally zipped as `.fb2.zip`) with embedded binary images, structured verse, and metadata sections.     |
| OPF   | The Open Packaging Format package document inside an EPUB: it declares metadata, the manifest of resources, and the spine.                        |
| Spine | The ordered list in the OPF that defines the linear reading order of an EPUB's content documents.                                                 |

## error-and-envelope-terms {#error-and-envelope-terms}

| Term                       | Definition                                                                                                                                                                                                         |
|----------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Result / AppError envelope | The uniform return shape `Result{data, error}` used across services; failures carry one typed `AppError` with an `ErrorCode` and a safe-details allowlist, enabling partial results. Defined in the `:api` module. |
| Round-trip golden test     | The automated test that asserts round-trip fidelity for representative documents of each supported format.                                                                                                         |
